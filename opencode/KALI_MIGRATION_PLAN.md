# Kali → MacBook Migration Plan

## Overview

Safe, phased migration of the OpenCode ecosystem from Kali Linux to macOS.
**Kali remains the lab/reference. MacBook is the primary workstation.**

## Migration Principles

1. **Never copy broken configs** — validate everything before importing
2. **Never copy secrets** — reconfigure API keys manually via .env
3. **Never copy caches/node_modules** — reinstall dependencies cleanly
4. **Always backup before import** — snapshot current MacBook state
5. **Validate after each step** — if something breaks, rollback immediately

## What NOT to Migrate ❌

| Item | Reason |
|---|---|
| `~/.config/opencode/opencode.json` | Contains `providers` (plural) — incompatible with OpenCode 1.16.2 |
| `~/.config/opencode/opencode.jsonc` | Contains `providers` key — incompatible format |
| Any file with `providers` (plural) | OpenCode 1.16.2 uses `provider` (singular) |
| `~/.local/share/opencode/opencode.db` | Session data, cache — will be regenerated |
| `~/.local/state/opencode/` | Runtime state — will be regenerated |
| `node_modules/` anywhere | Reinstall with `npm ci` or `npm install` |
| `package-lock.json` | Regenerate on MacBook |
| `.env` files with secrets | Recreate manually with new tokens |
| `~/.android/` cache | Will be regenerated by Android SDK |
| Gradle cache (`.gradle/`) | Will be regenerated on first build |
| Telegram session files | Re-authenticate on new machine |
| WhatsApp auth files | Re-authenticate on new machine |
| SSH keys | Do not copy — use existing MacBook keys |
| GPG keys | Do not copy — use existing MacBook keys |
| Any file with hardcoded IPs/paths | Paths differ between Kali and macOS |

## Migration Order

```
Phase A: Preparation (on MacBook)
  1. Run `ocbackup` — full snapshot of current state
  2. Run `ocdoctor` — verify MacBook is healthy
  3. Verify diff tools available: diff, meld, or vimdiff

Phase B: Review & Adapt (on Kali — read only)
  4. SSH into Kali or review files manually
  5. For each file in "What to Migrate":
     a. Read the file
     b. Identify macOS-specific changes needed
     c. Note any Kali-specific paths/commands

Phase C: Safe Copy (Kali → MacBook)
  6. Copy reviewed files one by one
  7. Never copy entire directories at once
  8. After each file:
     a. Validate syntax (JSON, shell, etc.)
     b. Adapt paths from Kali to macOS
     c. Run `ocdoctor` or relevant check

Phase D: Validation (on MacBook)
  9. Run `ocharness-check` — verify ecosystem structure
  10. Run `ocdoctor` — verify system health
  11. Run `ocbackup` — snapshot after migration
  12. Run OpenCode — verify it starts
```

## Safe Copy Commands

```bash
# From Kali, copy individual files securely:
# (Run on MacBook, pulling from Kali)

# Example: Copy a reviewed script
scp user@kali:~/harnesses/scripts/ocstart ~/harnesses/scripts/ocstart.kali-review
diff ~/harnesses/scripts/ocstart ~/harnesses/scripts/ocstart.kali-review

# Example: Copy documentation
scp user@kali:~/harnesses/docs/ANDROID.md ~/harnesses/docs/ANDROID.md.kali-ref

# NEVER do:
# rsync -avz user@kali:~/harnesses/ ~/harnesses/   # DANGEROUS — copies everything
```

## Rollback Points

| Step | Rollback Action |
|---|---|
| After Phase A (backup) | `cp ~/harnesses/backups/snapshot-* ~/harnesses/` |
| After any single file copy | Delete the file or restore from `ocsave` snapshot |
| After script import | `git checkout` or restore from backup |
| After config change | `cp ~/.config/opencode/opencode.json.bak.* ~/.config/opencode/opencode.json` |

## MacBook-Specific Adaptations

| Kali Path | macOS Path |
|---|---|
| `/home/user/` | `/Users/admin/` |
| `~/.local/share/opencode/` | `~/.local/share/opencode/` (same, but check) |
| `~/.config/opencode/` | `~/.config/opencode/` (same) |
| `/usr/bin/python3` | `/usr/local/bin/python3` (Homebrew) |
| `~/Android/Sdk` | `~/Library/Android/sdk` |
| `/usr/bin/adb` | `~/Library/Android/sdk/platform-tools/adb` |
| Linux line endings | macOS uses same LF — OK |
| `#!/bin/bash` | Same on macOS |
| `#!/usr/bin/env python3` | Same on macOS |
